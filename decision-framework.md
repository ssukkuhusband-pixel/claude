# 의사결정 프레임워크

이 문서는 기획팀의 모든 판단 기준을 정의한다. 어떤 기획적 결정이든 이 프레임워크를 따른다.

---

## 1. 의사결정 우선순위

```
데이터 > 논리 > 직감/다수결
```

- **데이터가 있으면 데이터를 따른다**
- **데이터가 없으면 논리로 빈틈 없이 따진다**
- **직감이나 다수결은 근거가 아니다**

---

## 2. 데이터 기반 판단

### 데이터가 존재하는 경우

어떤 시스템/기능을 도입할지 판단할 때:

1. 해당 시스템을 가진 **성공한 게임이 존재하는가?**
2. 그 게임의 **유저 리뷰/후기에서 해당 시스템에 대한 평가**는 어떤가?
3. 해당 게임이 **실제로 매출을 일으키고 있는가?**

위 3가지가 긍정적이면, 그 시스템은 좋은 시스템일 확률이 높다.

> **"내 생각엔 이래요"는 근거가 아니다. 데이터를 가져와라.**

### 데이터를 쪼개서 찾기

완전히 새로운 시스템(B+D)이라도, 구성 요소를 분리하면 데이터가 존재할 수 있다.

- B+D 시스템 → B의 데이터 + D의 데이터를 각각 확인
- 각 구성 요소의 유저 반응과 시장 성과를 종합하여 판단

---

## 3. 논리 기반 판단

### 데이터가 없는 참신한 시도의 경우

데이터가 없을 때는 논리로 검증한다. 다음을 반드시 수행:

1. **모든 경우의 수를 하나하나 따진다**
2. **예외 케이스에 대한 대비가 되어있는가 확인한다**
3. **논리적 허점이 없는지 확인한다**

### 다수결의 함정

- 여러 사람이 "좋네요"라고 해서 좋은 아이디어가 아니다
- 사람들은 아이디어의 **첫인상이 좋으면 숨어있는 허점이나 논리적 오류를 찾지 못한다**
- 다수의 긍정적 반응은 참고 자료일 뿐, 결정의 근거가 될 수 없다

---

## 4. 핵심 판단 체크리스트

어떤 기획적 판단이든, 아래 질문을 순서대로 적용한다.

### 체크 1: 시장 성공 가능성

- [ ] 이 게임/기능은 현재 모바일 시장에서 성공할 가능성이 높은가?
- [ ] 시장에 비슷한 것이 이미 존재하는가? 존재한다면 우리의 차별점은 무엇인가?

### 체크 2: 유니크함 검증

- [ ] 시장에서 본 적 없는 요소가 최소 1가지 이상 있는가?
- [ ] 대중이 "이건 다르다"고 느낄 수준(참신함의 임계점)을 넘었는가?
- [ ] 참신하지만 너무 난해하진 않은가? 설명을 들었을 때 바로 이해가 되는가?
- [ ] **이 유니크함을 15초 영상 하나로 전달할 수 있는가?**
- [ ] **같은 조합을 시도한 실패 사례가 있는가? 있다면 왜 실패했는가?**

### 체크 3: 재미 검증

- [ ] 성취감 요소가 명확한가? (성장, 수집, 실력 향상 등)
- [ ] 의외성 요소가 있는가? (뻔하지 않은 전개, 예측 불가한 변화)
- [ ] 성취감과 의외성의 템포가 적절한가? "조금 더 하고 싶다"를 유도하는가?
- [ ] **성취감의 리듬: 하나의 시스템/구간에서 최소 주 1회 이상 성취감을 느낄 수 있는가?**
- [ ] **한 세션 종료 시 유저에게 미완의 목표가 최소 1개 남아있는가? (선행 지표)**
- [ ] "다음 날 또 켜고 싶은가?" (결과 지표)

### 체크 4: 데이터/논리 검증

- [ ] 참고한 시스템의 시장 데이터(유저 반응, 매출, 성과)를 확인했는가?
- [ ] 데이터가 없는 영역은 논리적으로 모든 경우의 수를 따졌는가?
- [ ] 예외 케이스에 대한 대비가 되어있는가?
- [ ] "내 생각엔"이 아닌, 구체적 근거를 제시할 수 있는가?

### 체크 5: 리소스 효율

- [ ] 유니크한 핵심에 리소스가 집중되어 있는가?
- [ ] 핵심이 아닌 영역은 검증된 레퍼런스를 참고했는가?
- [ ] 컨셉이 통일되어 있는가? (게임 제목, 아트, UI, 시스템이 하나의 방향)

### 체크 6: BM 검증

- [ ] 유저가 "갖고 싶다"고 느낄 가치 있는 것이 존재하는가?
- [ ] 최종 목표까지의 단계가 설계되어 있는가?
- [ ] 각 단계에서 과금 포인트가 자연스럽게 배치되어 있는가?
- [ ] 적극적으로 판매하고 있는가?

### 체크 7: 장기 리텐션 검증

- [ ] 유저가 이 게임을 오래 할 이유가 명확하게 제시되어 있는가?
- [ ] "갖고 싶은 것" 또는 "성취하고 싶은 것"까지의 여정이 단계적으로 설계되어 있는가?
- [ ] 각 단계가 지치지 않고 진행할 수 있는 분량인가?

## 5. 피드백 시 적용법

팀원의 기획안을 리뷰할 때 이 프레임워크를 역순으로 적용한다.

### 참신함 영역에 대한 피드백

```
"이 시스템이 시장에서 참신하다는 근거는 무엇인가요?"
"비슷한 게임이 이미 존재하지 않나요?"
"참신하긴 하지만, 대중이 이해할 수 있는 수준인가요?"
```

### 참고 영역에 대한 피드백

```
"참고한 게임의 실제 성과 데이터가 있나요?"
"참고한 게임에서 이 시스템에 대한 유저 반응은 어떤가요?"
"해당 게임이 성공하지 못했다면, 이 시스템을 참고하는 것이 적절한가요?"
```

### 문서 품질에 대한 피드백

```
"이 케이스에서는 어떻게 동작하나요?" (예외 케이스 누락)
"이 설명이 X 페이지와 Y 페이지에 중복됩니다. 한 곳으로 합치세요."
"유저 화면에 어떻게 보이는지 시각적으로 표현해주세요."
```

---

## 6. 판단 시 주의사항

1. **"이건 재밌을 것 같아요"는 근거가 아니다** — 왜 재밌는지, 데이터는 무엇인지 제시해야 한다
2. **첫인상에 속지 마라** — 좋아보이는 아이디어일수록 허점을 더 꼼꼼히 찾아야 한다
3. **적당히 베끼지 마라** — 참신함의 임계점을 넘지 못하면 "비슷한 게임"으로 묻힌다
4. **리소스를 분산하지 마라** — 핵심이 아닌 곳에 힘을 쏟으면 핵심의 완성도가 떨어진다
5. **BM을 후순위로 미루지 마라** — 재미있지만 돈을 못 버는 게임은 실패한 게임이다
